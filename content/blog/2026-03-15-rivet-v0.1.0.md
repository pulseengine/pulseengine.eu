The faster you can produce code, the more important it becomes to prove *why* that code exists. rivet keeps the chain from requirement to test to evidence as YAML in git — machine-readable, agent-writable, validated in CI. When an auditor asks "show me the requirement for this function", the answer is a command, not a meeting.

## The problem

An AI agent can implement a feature, write the tests, and open a pull request in under ten minutes. What it does not do — what no agent does today — is maintain the traceability chain that safety standards demand. ISO 26262 does not care how fast you shipped. It asks: which requirement does this satisfy? Where is the test? Where is the evidence that the test passed on the version you released?

These questions are not optional in automotive, aerospace, or medical devices. They are the difference between a product that ships and one that does not.

Most teams track traceability in systems that live outside the codebase — databases, spreadsheets, proprietary ALM tools. That worked when humans wrote every artifact by hand. It breaks when agents produce code faster than anyone can update a separate tracking system. The traceability model drifts from the code within hours.

## rivet's answer

Keep the artifacts next to the code. Requirements, design decisions, safety analysis, test specifications — all as YAML files in the same git repository. rivet validates the links between them on every commit:

```yaml
- id: REQ-001
  type: requirement
  title: Component fusion preserves semantics
  status: approved
  links:
    - target: TST-001
      type: verifies
```

```bash
$ rivet validate
  Coverage: 100% (12/12 requirements traced to tests)
  Errors: 0
```

If an agent adds a feature without linking it to a requirement, CI fails. If a requirement exists without a test, rivet reports the gap. The traceability model is validated the same way the code is — automatically, on every push.

This matters for agents specifically: an agent can run `rivet add`, `rivet link`, and `rivet validate` as part of its workflow. The traceability chain becomes something agents maintain, not something humans reconstruct after the fact.

## What we use it for

We build a formally verified WebAssembly pipeline for safety-critical systems. That means every component in PulseEngine needs to carry its own traceability. rivet manages this — and it starts by managing itself:

- **rivet** tracks its own 344 artifacts, 71 features, 100% traceability coverage
- **gale** (our Zephyr RTOS kernel port) tracks 268 ASPICE artifacts at 97% coverage
- Each release publishes a compliance report to [pulseengine.eu/reports](/reports/)

rivet ships with schemas for STPA, Automotive SPICE, cybersecurity (TARA), and a lightweight dev schema. Pick what you need, combine them, validate.

## v0.1.0

22 CLI commands. `rivet validate` checks links, coverage, and schema rules. `rivet serve` runs an interactive dashboard. `rivet export --format html` produces a self-contained static report. `rivet impact REQ-001` shows everything downstream of a change. `rivet add` and `rivet link` let agents maintain artifacts from the command line.

402 tests. Cross-repository linking. Incremental validation. Kani proofs and Verus specs on core data structures.

```bash
git clone https://github.com/pulseengine/rivet
cd rivet && cargo build --release
./target/release/rivet validate
./target/release/rivet serve  # http://localhost:3000
```

The [repository](https://github.com/pulseengine/rivet) is the best example — it tracks its own development with the same schemas downstream projects will use.
